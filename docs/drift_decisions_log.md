**D-038 · 2026-07-23 — Cycle 03 audit remediation: the purity check judges every import by where it resolves, and body gameplay numbers move to `tune.ts`.**
The C3 audit of Cycle 03 found a **fourth** bypass of the brain/body check — the same family as the three before it (D-026 ×2, D-034): a *relative* path that tunnels into node_modules (`import from '../../node_modules/@babylonjs/core/...'`) slipped past, because the relative branch only checked for a body import, never the real package the path lands in. Rather than special-case a fourth shape, the check was **generalised to close the class**: every import — relative or bare, however written — is now judged by the package that owns the file it resolves to (`owningPackageOfFile`), and a resolved path inside node_modules is never recursed into, only identity-checked. All **five** bypass shapes (direct, transitive, computed-dynamic, npm-alias, relative-tunnel) reproduced and confirmed caught; the clean tree still passes. This is the generalization the checker had been circling for four cycles: stop trusting syntax, always ask where it resolves.
Also fixed, from the audit's notes: the tune law is now honoured in `src/body` — the player's collision radius, the per-kind obstacle footprints, the decorative-instance radii, the pond tap forgiveness, and the tap-vs-drag gesture thresholds moved from hard-coded body literals into `tune.ts` (they gate reachability and interaction, exactly the class D-026 established belongs there); the HUD's "low" vital thresholds now read the same `TUNE.*LowHintAt` constants the hint system uses, so the bar's cue and the hint can never disagree; `save.ts` clamps every vital to its valid range on load, so a corrupt or hand-edited save can no longer carry a negative health that reconcile would hold negative forever (a health of 0 self-heals to a respawn on the next tick); and the device harness takes the death screenshot after confirming the overlay is up. *Status: standing.* — C2

**D-034 · 2026-07-23 — Cycle 02 audit remediation: the purity check resolves real package identity.**
The C3 audit of Cycle 02 found a *third* bypass of the brain/body check, in the same family as Cycle 01's two: an npm alias (`"x": "npm:@babylonjs/core"`) lets a brain file `import from 'x'` and load a renderer straight past a forbidden-*name* list. Root cause across all three: the check trusted the name the code typed. Fixed at the root — `tools/check-purity.mjs` now resolves every bare specifier to the file it actually loads, walks up to that file's owning `package.json`, and judges the **real declared package name**, so an alias resolves to `@babylonjs/core` and is caught. All four bypass classes (direct import, transitive re-export, computed dynamic import, npm alias) reproduced and confirmed caught (exit 1); the clean tree still passes. Two audit notes also fixed: the `+2 hours` fire-feed feedback now reads `TUNE.fireBurnGameHoursPerWood` live rather than hard-coding it, and the harness's "within seconds" check tightened from 45 s to 8 s. Added beyond the notes: the device harness now **fails** if it finds itself on a software renderer without `--software` having asked for it, so a GPU-less runner can never misread an 8 fps SwiftShader number as a real A3 pass. *Status: standing.* — C2

**D-033 · 2026-07-23 — The frame rate is traced beside the save, not inside it.**
Cycle 02's spec extends the playtest trace with an FPS median, but A1 requires the brain's `TraceState` to stay byte-identical across the pivot — and the brain has no opinion on frame rate anyway. **D-032 · 2026-07-23 — The HUD and panels are DOM over the canvas, not engine-drawn; plus a scene debug handle.**
Text in the DOM is crisp at any device pixel ratio, costs the GPU nothing, scales with the player's own font settings, and is reachable by a screen reader — all of which §I.18 rule 7's accessibility baseline asks for and none of which comes free from text drawn into a 3D scene. It also keeps the frame budget for the island. Consequence: every tappable control is a real `<button>`, so the harness clicks them as a user would rather than guessing at pixels.
Alongside it the build exposes `window.__driftScene`. It is more surface than D-022's narrow save accessor, and it is kept deliberately: it turned two silent rendering bugs into five-minute diagnoses this cycle — an entire invisible beach (back-face-culled terrain) and a fire with no flame — where reading the code had produced only plausible wrong answers. Revisit if anything the player could exploit ever depends on it. *Status: standing.* — C2

**D-026 · 2026-07-22 — Audit remediation: the purity check now walks the import graph, and two hit-test numbers moved into `tune.ts`.**
The C3 audit of Cycle 01 (PASS-WITH-NOTES) demonstrated that `tools/check-purity.mjs` only regexed the brain's own files: a brain module importing Phaser *through* an intermediary in `src/data/` passed the check. The brain/body law is the project's one non-negotiable rule (§II.5) and its enforcement was weaker than it looked, so the check was rewritten to resolve and walk the brain's whole transitive import graph and to print the offending chain; the bypass was reproduced, then confirmed caught (exit 1). Also moved: the fire's tap radius (was a literal `40`) and the wood-node tap slack (was in the presentation table) into `tune.ts` as `fireTapRadius` and `nodeTapSlack` — both widen hit targets, so by D-023's own test they change how the game *plays*, not how it looks. The audit's remaining notes (an over-broad test title, two harness checks that could not fail, and a conditional check that could silently skip) were all fixed rather than accepted.
**Second pass, same audit:** C3 then found a *new* bypass on the rewritten checker — `import(parts.join(''))`, a dynamic import with a computed specifier, is working Phaser-loading code that no static analysis can follow. Closed by treating any non-literal `import()`/`require()` inside the brain's closure as a violation in itself: the brain has no legitimate use for one, so the escape hatch is the defect. Reproduced and confirmed caught. C3 also found that the device harness's fire-burn check was flaky (1 failure in 3 live runs) because it compared fuel against a *nominal* three-minute absence while the harness's own reloads leaked real seconds into it — so the check was measuring the harness's punctuality, not the game's arithmetic. Rewritten to assert the invariant instead (whatever time actually passed, the fire burned it at the tuned rate), which tightened the tolerance from ±0.02 to ±1e-6 and made it exact; three consecutive runs, 30/30. Last bare coefficient (`interactRadius * 0.6`) moved to `TUNE.approachStopFraction`. *Status: standing.* — C2

**D-025 · 2026-07-22 — The offline fairness floor is a rule about ABSENCE, not about elapsed time.**
The C3 audit observed that the D-011 warmth floor is gated on a single `reconcile` call reaching the report threshold, so the frame loop's millisecond spans never trigger it: a player who leaves the app gets the floor, a player who sits idle with the screen on does not. Reviewed and **kept deliberately**, because it is the correct reading of D-011: mercy exists for time the player could not act in. Presence is agency; absence is not. Someone watching the fire go out and choosing to do nothing has made a decision, and the charter's fair-challenge test (§I.18 rule 2) is satisfied — it is telegraphed, understandable, preventable, and recoverable. The behaviour is now documented at the gate in `reconcile.ts` and pinned by a test, so it cannot drift silently. **One dependency named explicitly, at C3's insistence:** on the primary device this rule leans on the phone's OS auto-lock, which typically fires inside two minutes of no touches and takes the correctly-floored `visibilitychange` path — so on Android the exposed window is narrow. On a desktop browser (which the charter permits; Android is primary, not exclusive) there is no auto-lock safety net and the gap is fully real. The decision is sound but it is not self-supporting, and Cycle 02 must weigh it on that basis rather than on "presence is agency" in the abstract.
**Flagged for Cycle 02**, where D-011's frame is already the design topic: once warmth carries real stakes, confirm this or replace it with a floor keyed to time since the last interaction. *Status: standing; revisit in C02.* — C2

**D-022 · 2026-07-22 — The device acceptance checks are automated (`tools/smoke.mjs`), with a local debug hook.**
"It plays right on the phone" (Charter §II.4) was the one gate nobody could re-run. `tools/smoke.mjs` drives an already-installed Chrome in Android emulation with real touch events and asserts against live game state: 31 checks covering A3, A4, A6 and the observable half of A7, re-runnable against any deployed URL — which is what lets C3 audit a build instead of trusting a claim. **D-021 · 2026-07-22 — The placeholder audio cues are synthesised in-repo, not sourced.**

**D-020 · 2026-07-22 — One input path: every tap routes through a single scene-level handler.**
Found during the build: Phaser's per-Game-Object input fired reliably for mouse but **not** for touch in this configuration — every button in the game was dead under a thumb while working perfectly on a desktop, and nothing in the code looked wrong. Rather than carry two input paths and hope, tappable things register a rectangle with `src/body/ui/Taps.ts` and the scene's one `pointerdown` handler routes to the topmost enabled target. One path, one place to debug, and no button can be silently un-hittable. This class of bug is exactly what the device smoke test (D-022) exists to catch. *Status: standing.* — C2

**D-011 · 2026-07-22 — Offline fairness constitution (charter §II.10; frames the Cycle 02 spec).**
Supersedes the "offline-drain gentleness" open note.
